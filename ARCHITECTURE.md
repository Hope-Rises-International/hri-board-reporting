## 1. What This Service Does

`hri-board-reporting` is a GitHub Pages static website that hosts monthly board updates for Hope Rises International's Board of Directors. Each month, a new self-contained HTML report is authored (or generated with AI assistance) and pushed to the repository. The live site at `https://hope-rises-international.github.io/hri-board-reporting/` serves a card-based landing page (`index.html`) that links to each monthly report by filename (e.g., `board-update-april-2026.html`). There is no backend, no database, no login, and no dynamic code — the site is pure HTML and CSS. Board members access reports via the direct link; a `robots.txt` that blocks all crawlers prevents search engines from indexing the content.

| **Schedule** | None — only updates when content is pushed |

## 7. What Breaks If This Stops Running

Because this repo has no automated pipeline, "stops running" means the site becomes stale or unreachable — not a process failure.

- **If GitHub Pages goes down:** Board members cannot access reports. No data is lost; the HTML files remain in the repo. Service resumes automatically when GitHub Pages recovers.
- **If no one updates the repo:** Board members see the last-published report. There is no auto-expiry or error state — the old content simply stays up indefinitely.
- **Business impact of staleness:** Board members miss current financial and program updates. This is a communication gap, not a financial system failure — no downstream systems depend on this site.
- **If Google Fonts CDN is unreachable:** The site still loads and is fully readable; the browser falls back to a system sans-serif font. Visual branding is degraded but content is unaffected.

## 8. Three Most Likely Failure Modes

### Failure Mode 1: Report is pushed but the landing page card is not updated

**What goes wrong:** A new monthly HTML file (e.g., `board-update-may-2026.html`) is committed and pushed, but no new card is added to `index.html`. The new file is technically accessible by direct URL but is invisible from the landing page.

**Symptom:** Board members see the landing page with only older cards. The new month's report does not appear. Someone who knows the direct URL can still reach it.

**How to fix:** Edit `index.html` — add a new `<a class="card" href="board-update-may-2026.html">` block at the top of the `.cards` div (newest first), commit, and push. GitHub Pages rebuilds within ~90 seconds.

---

### Failure Mode 2: GitHub Pages build silently fails or gets stuck

**What goes wrong:** A push is made but GitHub Pages does not rebuild. This can happen if the repository Pages setting is misconfigured (e.g., source branch was changed from `main`), or if GitHub has a Pages outage.

**Symptom:** The live site shows stale content after a push. The commit appears in the GitHub repo history but the URL still serves the old version after several minutes.

**How to fix:** Go to `https://github.com/Hope-Rises-International/hri-board-reporting/settings/pages` and confirm the Source is set to "Deploy from a branch," branch = `main`, folder = `/ (root)`. If correct, check the Actions tab for a Pages deployment workflow error. If GitHub is having an outage, wait and re-push. If the Pages setting was wrong, correct it and push a trivial whitespace commit to trigger a rebuild.

---

### Failure Mode 3: Old `index.html` reports are accidentally overwritten

**What goes wrong:** A contributor follows the original (pre-April 2026) update model and replaces `index.html` with a full board report instead of creating a new dated file and adding a card.

**Symptom:** The landing page disappears and is replaced with a single full report. Previous months' report links on the landing page are gone (though their HTML files still exist in the repo).

**How to fix:** Run `git log --oneline` to identify the bad commit. Run `git revert <commit-hash>` to create a new revert commit (do not use `git reset --hard` as that rewrites history on a public repo). Push the revert. Then re-apply the correct change — add the new dated file and update the landing page card list.

## 9. How to Manually Re-Run (Publish a New Board Update)

There is no automated pipeline to re-trigger. "Re-running" means manually publishing a new or corrected report.

**Step 1 — Clone the repo (if not already cloned locally):**
```bash
git clone https://github.com/Hope-Rises-International/hri-board-reporting.git
cd hri-board-reporting
```

**Step 2 — Create the new monthly HTML file:**

Copy the structure of an existing report as your starting point:
```bash
cp board-update-april-2026.html board-update-may-2026.html
```
Edit `board-update-may-2026.html` with the new month's content. The title tag, header label, and all body copy should be updated.

**Step 3 — Add a card to the landing page:**

Edit `index.html`. Inside the `<div class="cards">` block, add a new card at the top (newest first):
```html
<a class="card" href="board-update-may-2026.html">
  <span class="card-tag">May 2026</span>
  <h2>[Headline for this month's report]</h2>
  <p>[One or two sentence summary for the card.]</p>
  <span class="read-more">Read update &rarr;</span>
</a>
```

**Step 4 — Commit and push:**
```bash
git add index.html board-update-may-2026.html
git commit -m "Add May 2026 board update"
git push origin main
```

**Step 5 — Verify deployment:**

Wait 60–90 seconds, then open `https://hope-rises-international.github.io/hri-board-reporting/` in a browser. Confirm the new card appears and the link opens the correct report.

**To fix a typo in an already-published report:**

Edit the relevant HTML file, commit, and push. GitHub Pages will re-serve the updated file within ~90 seconds. No other steps required.
